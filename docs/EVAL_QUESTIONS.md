## 프로젝트 전체

### 이 서비스가 해결하는 문제는 무엇인가?

CareerFit AI는 학생이 자신의 전공, 보유 스킬, 관심 직무를 입력했을 때 실제 취업·공모전 데이터를 기반으로 현재 역량을 분석하고, 어떤 공고와 연결되는지, 어떤 역량을 보완해야 하는지 알려주는 서비스이다.

기존에는 학생이 공고를 직접 찾아보고 요구 역량을 하나씩 비교해야 했지만, 이 서비스는 공고 데이터를 기반으로 추천 공고와 부족한 역량을 자동으로 정리해준다.

### RAG를 사용하지 않으면 어떤 문제가 생기는가?

RAG를 사용하지 않으면 AI가 실제 공고 데이터를 참고하지 않고 일반적인 조언만 할 가능성이 높다.  
이 경우 추천 공고나 보완 역량의 근거가 약해지고, 실제 데이터와 맞지 않는 답변이 나올 수 있다.

RAG를 사용하면 ChromaDB에서 관련 공고를 먼저 검색하고, 그 결과를 바탕으로 AI가 답변하기 때문에 분석 근거가 더 명확해진다.

### 가장 어려웠던 구현은 무엇이고, 어떻게 해결했는가?

가장 어려웠던 부분은 FastAPI 백엔드, ChromaDB 기반 RAG 검색, Gemini API, React 프론트엔드를 하나의 흐름으로 연결하는 과정이었다.

처음에는 백엔드 API 응답과 프론트엔드 출력 구조를 맞추는 과정에서 혼동이 있었지만, `/analyze` 응답에 `answer`와 `sources`를 함께 포함하도록 정리하고, React에서 결과 카드와 출처 카드를 분리해서 출력하는 방식으로 해결했다.

또한 Docker와 Render 배포 과정에서는 `.env`와 환경변수 관리가 중요했기 때문에 실제 API Key는 GitHub에 올리지 않고, 로컬에서는 `.env`, Render에서는 Environment Variables로 관리하도록 구성했다.

## 데이터 관련

### 왜 이 데이터를 선택했는가?

취업·공모전 데이터는 사용자의 전공, 스킬, 관심 직무와 실제 요구 역량을 비교하기에 적합한 데이터이다.

특히 CareerFit AI의 목적이 “포트폴리오 방향 추천”이기 때문에, 단순한 일반 조언보다 실제 공고 데이터에 기반한 분석이 더 설득력 있다고 판단했다.

### 데이터의 어떤 부분이 신뢰할 수 없을 수 있는가?

현재 데이터는 실제 대규모 채용 플랫폼 전체를 반영한 것이 아니라, 프로젝트 실습을 위해 구성한 제한된 공고 데이터이다.  
따라서 공고 수가 적거나 특정 직무에 편중될 수 있고, 실제 기업의 최신 채용 조건과 다를 수 있다.

또한 공고의 스킬 키워드가 사람이 정리한 형태라면 표현 방식이 통일되지 않거나 일부 누락될 가능성도 있다.

### SQLite와 ChromaDB에 같은 데이터를 저장하는 이유는 무엇인가?

SQLite와 ChromaDB는 같은 데이터를 저장하더라도 목적이 다르다.

SQLite는 공고 데이터를 표 형태로 구조화해서 저장하고, 특정 컬럼이나 조건에 따라 조회하기 위한 용도이다.  
반면 ChromaDB는 공고 내용을 문서 형태로 변환한 뒤 의미 기반 검색을 수행하기 위한 용도이다.

즉 SQLite는 정형 데이터 관리에 적합하고, ChromaDB는 RAG 검색에 적합하기 때문에 같은 데이터를 서로 다른 목적에 맞게 저장했다.

### 결측치와 중복을 어떻게 처리했는가?

Pandas를 사용해 CSV 데이터를 읽은 뒤, 비어 있는 값이나 중복 데이터를 확인했다.

결측치는 분석에 필요한 주요 컬럼을 기준으로 확인하고, 필요한 경우 빈 문자열이나 기본값으로 처리했다.  
중복 데이터는 같은 공고가 여러 번 들어가지 않도록 확인하고 제거했다.

이를 통해 RAG 문서로 변환할 때 불필요한 중복 검색 결과가 나오지 않도록 했다.

## RAG 관련

### RAG 구조에서 ChromaDB는 어떤 역할을 하는가?

ChromaDB는 RAG 구조에서 벡터 데이터베이스 역할을 한다.

공고 데이터를 문서 형태로 변환해 ChromaDB에 저장하고, 사용자가 전공·스킬·관심 직무를 입력하면 그 내용과 의미적으로 가까운 공고를 검색한다.

이렇게 검색된 공고는 Gemini가 답변을 생성할 때 참고 자료로 사용되며, 프론트엔드에서는 RAG Sources로 함께 표시된다.

### n_results 값을 높이면 어떤 트레이드오프가 생기는가?

`n_results` 값을 높이면 더 많은 공고를 검색 결과로 가져올 수 있기 때문에 다양한 후보를 참고할 수 있다.

하지만 너무 많은 문서를 가져오면 관련성이 낮은 공고까지 포함될 수 있고, AI 답변이 산만해질 수 있다.  
또한 프롬프트에 들어가는 문서 양이 늘어나 처리 비용과 응답 시간이 증가할 수 있다.

따라서 적절한 개수의 검색 결과를 선택하는 것이 중요하다.

### 검색 결과 품질을 어떻게 검증했는가?

직접 전공, 스킬, 관심 직무를 입력해보고, 반환된 RAG Sources가 입력 내용과 관련 있는지 확인했다.

예를 들어 전자전기컴퓨터공학부, Python, C, 전자회로, SPICE, 반도체 회로 설계와 같은 입력을 넣었을 때, 반도체 소자·회로 시뮬레이션 엔지니어와 아날로그 IC 설계 엔지니어가 출처로 나오는지 확인했다.

또한 AI 분석 결과가 RAG Sources에 나온 공고 내용을 근거로 설명하는지도 함께 점검했다.

### Hallucination이 무엇이며, RAG가 이를 어떻게 줄이는가?

Hallucination은 AI가 실제 근거가 없거나 사실과 다른 내용을 그럴듯하게 만들어내는 현상이다.

RAG를 사용하면 AI가 답변을 생성하기 전에 관련 문서를 먼저 검색하고, 그 문서 내용을 기반으로 답변하게 된다.  
따라서 완전히 일반적인 추측보다 실제 데이터에 근거한 답변을 만들 수 있어 Hallucination 가능성을 줄일 수 있다.

## API 관련

### /analyze 엔드포인트가 POST를 사용하는 이유는 무엇인가?

`/analyze`는 사용자의 전공, 보유 스킬, 관심 직무와 같은 입력 데이터를 서버로 전달해야 한다.

이 데이터는 단순 조회용 URL 파라미터보다 요청 본문에 담아 보내는 것이 적절하기 때문에 POST 방식을 사용했다.  
또한 분석 요청은 서버에서 AI 응답을 생성하는 작업이므로 GET보다 POST가 더 적합하다.

### 응답 JSON에 sources를 포함시킨 이유는 무엇인가?

`sources`를 포함하면 AI가 어떤 공고 데이터를 참고했는지 사용자에게 보여줄 수 있다.

이를 통해 AI 분석 결과가 단순한 추측이 아니라 실제 공고 데이터를 근거로 생성되었다는 점을 설명할 수 있다.  
또한 발표 시 RAG 구조를 보여주는 핵심 근거로 활용할 수 있다.

### /health 엔드포인트가 왜 필요한가?

`/health` 엔드포인트는 서버가 정상적으로 실행 중인지 빠르게 확인하기 위한 API이다.

로컬 실행, Docker 실행, Render 배포 후 서버 상태를 확인할 때 유용하다.  
특히 배포 환경에서는 `/health`를 통해 서비스가 정상적으로 떠 있는지 확인할 수 있다.

## 보안 관련

### .env 파일을 GitHub에 올리면 어떤 일이 생기는가?

`.env` 파일에는 Gemini API Key 같은 민감한 정보가 들어갈 수 있다.

이 파일이 GitHub에 올라가면 API Key가 외부에 노출될 수 있고, 다른 사람이 내 API Key를 사용해 비용이 발생하거나 보안 문제가 생길 수 있다.

따라서 `.env`는 반드시 `.gitignore`에 포함하고, 실제 Key는 GitHub에 올리지 않아야 한다.

### .env와 .env.example의 차이는 무엇인가?

`.env`는 실제 실행에 사용하는 환경변수 파일이고, 실제 API Key가 들어간다.  
이 파일은 로컬에서만 사용하고 GitHub에 올리면 안 된다.

`.env.example`은 어떤 환경변수가 필요한지 알려주는 예시 파일이다.  
실제 Key 대신 `your_gemini_api_key_here` 같은 예시값만 넣으며, GitHub에 올려도 된다.

## Docker 관련

### Docker를 사용하는 이유는 무엇인가?

Docker를 사용하면 개발 환경과 실행 환경을 하나의 컨테이너로 묶을 수 있다.

로컬 컴퓨터마다 Python 버전이나 패키지 설치 상태가 달라도, Docker 이미지를 사용하면 동일한 환경에서 FastAPI 서버를 실행할 수 있다.  
또한 Render 같은 배포 환경에서도 Dockerfile을 기반으로 쉽게 서비스를 배포할 수 있다.

### Dockerfile에서 EXPOSE와 CMD의 역할은 무엇인가?

`EXPOSE`는 컨테이너 내부에서 어떤 포트를 사용할지 명시하는 역할을 한다.  
예를 들어 FastAPI 서버가 8000번 포트에서 실행되면 `EXPOSE 8000`으로 표시할 수 있다.

`CMD`는 컨테이너가 실행될 때 자동으로 수행할 명령어를 정의한다.  
이 프로젝트에서는 컨테이너 실행 시 `uvicorn main:app` 명령으로 FastAPI 서버가 시작되도록 설정했다.
